# GitHub 开源参考实现检索协议

> 这是通用版 V2 的核心差异化能力。Agent 在完成 S1-S6 分析后，对高 AI 可行性的工作颗粒度，
> 主动从 GitHub 开源社区检索成熟参考实现，打通"应该做什么"到"用什么做"的最后一公里。

---

## 一、触发门控（Gate Check）

对每个工作颗粒度，**必须同时满足以下两个条件**才触发搜索：

### 条件 1：S4 执行主体标签
S4 标签 ∈ {**自动接管**, **AI主辅协同**}

### 条件 2：S6-B 约束通过
至少满足以下 3 项中的 2 项：
- 技术成熟度 ≥ **中**
- 市场供给状态 ≠ **空白**
- 成本等级 ≤ **中**

### 不触发的情况
- S4 = 人工主导，AI辅助 或 人工主导，AI仅弱辅助
- S6-B 三项中两项以上为不利（技术低/市场空白/成本高）
- Agent 应明确告知用户："这类工作目前还不适合直接找开源实现，更适合______方式"

---

## 二、搜索查询构建

### Step 1：特征提取

从颗粒度 S2 记录中提取以下字段：
- 颗粒度名称（中文）
- 颗粒度定义（一句话）
- 典型场景
- 当前痛点
- S6-A 能力标签（如 A:读取理解, D:分类分级, E:生成表达）

### Step 2：英文关键词翻译

将颗粒度名称和定义翻译为英文关键词。翻译策略：
- 动宾结构 → 英文动名词/名词短语
- 例如："会议纪要自动生成" → "meeting notes summarization", "meeting minutes automation"
- 例如："客户邮件分类与路由" → "email classification routing", "ticket triage automation"

### Step 3：构建 3 类并行查询

| 查询类型 | 语言 | 模板 | 示例 |
|----------|------|------|------|
| **语义查询** | 中文 | `{颗粒度描述} {痛点关键词} 开源 工具 GitHub` | `会议纪要自动生成 AI摘要 开源工具` |
| **技术查询** | 英文 | `{english keywords} open source tool github` | `meeting notes summarization open source tool` |
| **能力查询** | 英文 | `{S6-A capability} automation {domain} github` | `text summarization pipeline automation github` |

> 3 类查询应**并行执行**，使用 WebSearch 工具，每条查询自动添加 `site:github.com` 限定。

## 三、搜索结果处理

### Step 4：去重与合并

- 以 GitHub repo URL（`github.com/{owner}/{repo}`）为主键去重
- 同一 repo 出现在多个查询中 → 保留得分最高的那次匹配
- 合并后得到候选列表

### Step 5：成熟度评分矩阵

每个候选 repo 按 4 轴评分，满分 12 分：

| 维度 | 权重 | 0 分 | 1 分 | 2 分 | 3 分 | 4 分 |
|------|------|------|------|------|------|------|
| **Stars** | 重要 | < 50 | 50-500 | 500-2k | 2k-10k | > 10k |
| **最近提交** | 关键 | > 2 年 | 6月-2年 | 1-6月 | < 1月 | — |
| **文档质量** | 重要 | 仅 README 标题 | 基础 README | 有文档站点/详细 README | 完整文档+示例 | — |
| **社区活跃** | 参考 | 无 Issue/PR 活动 | 偶有活动 | 活跃（周级） | — | — |

**评分说明**：
- **主数据源为 WebSearch 摘要**：Stars 数和最近提交日期优先从 WebSearch 返回的摘要中提取。大部分搜索结果的摘要行包含 Star 数和最后更新时间。
- **WebFetch 作为补充（可能被阻止）**：如果 WebSearch 摘要信息不足以评分（如缺少文档质量、社区活跃度信息），尝试用 WebFetch 获取 repo 首页。**注意：github.com 可能在某些环境中被 WebFetch 阻止**——此时直接基于 WebSearch 摘要和搜索结果片段进行评分，并在呈现时标注"评分基于搜索结果摘要，精度可能受限"。
- **文档质量**：从 README 内容的搜索结果片段判断（是否有安装说明、使用示例、API 文档）
- **社区活跃度**：从搜索结果的 Issue/PR 讨论片段判断
- **评分不确定时，宁可低估一档**：如果无法确定 Stars/最近提交/文档/社区的精确值，取可确认的最低分。成熟度分级宁可偏保守（标 🟡 而不标 🟢），避免过度承诺。

### Step 6：成熟度分级

| 总分 | 成熟度 | 标识 | 含义 |
|------|--------|------|------|
| ≥ 10 | 🟢 生产可用 | Production-ready | 有大规模使用案例，文档齐全，持续维护 |
| 7-9 | 🟡 活跃开发 | Active development | 功能可用，社区活跃，但可能缺文档或案例 |
| < 7 | 🔴 实验阶段 | Experimental | 概念验证或早期开发，稳定性不确定 |

### Step 7：采纳难度评估

| 难度 | 标识 | 判断标准 |
|------|------|----------|
| 直接可用 | 🟢 | SaaS/托管服务，注册即用；或 Docker 一键部署 |
| 需要配置 | 🟡 | 需要本地部署+参数调优+数据准备，预计 < 1 周 |
| 需要二次开发 | 🔴 | 只有核心库/框架，需要自行开发业务逻辑 |

## 四、推荐呈现格式

### 每颗粒度推荐卡片

```markdown
### 🔍 颗粒度：[颗粒度名称]

**当前痛点**：[从 S2 记录提取，1 句]
**所需 AI 能力**：[从 S6-A 提取，如 文本生成(E) + 分类分级(D)]
**检索关键词**：`语义查询所用词` / `技术查询所用词` / `能力查询所用词`

| # | 项目 | Stars | 成熟度 | 采纳难度 | 映射说明 |
|---|------|-------|--------|----------|----------|
| 1 | **[项目名]** github.com/owner/repo | X.Xk | 🟢 生产可用 | 🟡 需要配置 | [1-2 句：该项目的 X 功能可直接覆盖你的 Y 场景] |
| 2 | **[项目名]** github.com/owner/repo | XXX | 🟡 活跃开发 | 🟢 直接可用 | [1-2 句] |
| 3 | **[项目名]** github.com/owner/repo | XXX | 🟡 活跃开发 | 🔴 需要二次开发 | [1-2 句] |

#### 💡 建议采纳路径

**优先评估 [#1 项目名]**：[2-3 句具体建议，说明为什么优先、从哪开始、预期效果]

**备选方案**：如果 #1 因 [某约束] 不适用，可考虑 [#2 项目名] 的 [某特性]

**如果自行开发**：[#3 项目名] 的核心库可以作为起点，减少从零开发的工程量。

> ⚠️ 以上建议基于公开信息检索生成，未进行实际测试验证。建议在实际采纳前由技术团队进行可行性评估和 PoC 验证。
```

### 多颗粒度情况

如果有多个颗粒度触发搜索，按以下优先级排列：
1. S4 = 自动接管 的颗粒度优先于 AI主辅协同
2. 相同 S4 标签下，S5 = 提效 的优先于其他（容易出成果）
3. 每个颗粒度独立输出推荐卡片
4. 如果超过 3 个颗粒度触发，Agent 先列总览表再逐个展开

## 五、搜索失败处理

### 情况 A：三个查询都没有找到明显匹配

Agent 应：
1. 告知用户："我用 [列出关键词] 检索了 GitHub，暂时没找到直接匹配的开源项目。"
2. 给出可能的原因：
   - 这个领域可能太垂直/太小众
   - 开源方案可能用了不同的命名方式
   - 可能还没有成熟的开源实现
3. 提供**分层替代建议**（按从快到慢的顺序）：
   - **先用通用工具凑**："虽然没有专门做 [颗粒度] 的开源项目，但 [通用工具名] 的 [某功能] 可以覆盖核心需求"（如：低代码表单工具、通用 BI 平台、RPA 工具）
   - **试试 MCP Server / 插件生态**："可以在 Claude Code / Cursor 等 AI 工具中搜索 [关键词] 相关的 MCP Server 或插件"
   - **基于通用能力组合自定义**："我可以帮你设计一个基于通用能力组合的实现方案——用 [能力 A] + [能力 B] 组合来覆盖你这个场景"
   - **换一批关键词再搜**："你可以试试用 [建议的替代关键词] 继续搜索，有时候换个说法就能找到"

### 情况 B：找到的 repo 都很新/Star 很低

Agent 应：
1. 标注为 🔴 实验阶段
2. 评估采纳风险："这些项目都还比较早期，Star 和文档都不够，建议先观望或只做技术验证"
3. 给出替代路径：
   - "如果要现在就落地，可以考虑用 [更成熟的相关领域工具] 做适配"
   - "也可以用低代码/无代码平台（如简道云、明道云、Zapier、n8n）先搭一个 MVP，验证流程跑通后再考虑自研或等开源方案成熟"

### 情况 C：找到很多但都不够精准

Agent 应：
1. 从结果中挑最接近的 2-3 个
2. 说明为什么不完美匹配（"它的核心功能是 X，而你需要的是 Y"）
3. 建议："可以作为参考架构，但需要做 [具体改造方向]"

### 情况 D：找到 1-2 个匹配但供给明显偏少 🆕

常见于垂直行业/小众领域。Agent 应：
1. 如实呈现已找到的项目，同时标注"该领域开源供给偏少"
2. 主动建议替代实现路径：
   - **相邻领域工具移植**："虽然没有 [精确领域] 的工具，但 [相邻领域] 的 [项目名] 经过配置后可以覆盖 70% 的需求"
   - **低代码平台 + 模板**："像 [简道云/明道云/Retool/Appsmith] 这类低代码平台可以快速搭建 [表单/仪表板/流程] 而不需要从零写代码"
   - **通用 SaaS 工具的垂直用法**："[Notion/Airtable/飞书多维表格] 这类通用工具可以通过模板化配置变成 [你的场景] 的专用工具"
3. 给用户一个务实的选择："你是想先试试已找到的这 1-2 个项目，还是我帮你用低代码/通用工具设计一个替代方案？"

## 六、Agent 执行注意事项

1. **不要编造 repo**：只推荐搜索实际返回的项目。如果搜不到就说搜不到。
2. **不要过度承诺**：搜索结果是参考，不是保证。始终加 ⚠️ 免责声明。
3. **映射说明是核心价值**：只说"这个 repo 很火"没用，必须说清楚"它的 X 怎么对应你的 Y"。
4. **控制数量**：每颗粒度最多 5 个推荐，最少 1 个（如果只有 1 个匹配）。0 个匹配时按"情况 A"处理。
5. **优先中文友好**：如果有中文 README 的项目，优先推荐（降低用户使用门槛）。
6. **考虑合规**：注意 License 类型（MIT/Apache 优先，GPL 标注风险）。
