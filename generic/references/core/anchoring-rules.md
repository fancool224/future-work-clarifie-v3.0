# 通用版锚定规则

> 共创过程中的 8 条防漂移规则。Agent 在每一步判断时都必须检查这些规则。

---

1. 先拆工作颗粒度，再谈 AI 影响。不先拆开就直接下"这个岗位会不会变"的结论。

2. 先判断问题处理模式（S3），再讨论谁来做（S4）。问题的性质决定分工，不能反过来。

3. 不要对本质上不确定的问题强行给出确定答案。风险型问题给等级和倾向，设计型问题给路径和选项。

4. 低阶执行工作最容易被接管，高阶设计工作最容易发生价值重构。不要把"AI 替代执行"和"工作价值消失"混为一谈。

5. 不是所有变化都等于替代。很多变化发生在工作形态、节奏、质量维度上，而不是"人还是机器"的二选一。

6. 一旦某颗粒度涉及多个角色/部门/职能的共同判断，就把它视为跨角色关键议题，不适合只从单一视角分析。

7. 工作未来命题必须收敛成"从什么转向什么"的句式。不能停留在"会变化"的模糊结论上。

8. 对 S4=自动接管 或 AI主辅协同 且 S6-B 约束通过的颗粒度，必须主动检索 GitHub 开源参考实现。不能只给"可以做"的判断而不给"用什么做"的路径。

---

## Agent 自检清单

每轮判断结束后，Agent 自查：

- [ ] 我有没有跳过颗粒度拆解直接下结论？
- [ ] 我有没有对高风险/高不确定性问题给出了过于确定的口吻？
- [ ] 我有没有把"AI 替代执行"表述为"工作消失"？
- [ ] 我有没有遗漏跨角色的判断视角？
- [ ] 我的结论是否可以收敛成"从 X 转向 Y"？
- [ ] 对于可 AI 化的颗粒度，我有没有主动检索开源参考实现？
