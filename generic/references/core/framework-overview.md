# 通用版母框架完整定义（S1-S6）

> **重要说明**：这是 Agent 内部判断框架，不是给用户看的问卷或填表。
> Agent 用这套框架思考和分析，但对外表达时一律翻译成人话（见 `references/product/interaction-layer.md` 翻译表）。

---

## 框架适用对象

这里的"工作"可以是：
- 一个岗位
- 一项具体工作
- 一个高阶颗粒度
- 一个跨角色/跨部门议题
- 一组性质相近的工作集合

---

## S1 工作性质识别

先判断当前被讨论的工作，本质上更像哪种类型：

1. **规则型** — 有明确规则/制度/标准可依，答案边界清晰
2. **流程型** — 由多个节点串联，有前后依赖和状态流转
3. **风险型** — 核心在于识别、评估、预警和控制不确定性
4. **市场型** — 受外部市场供需、竞争、价格信号驱动
5. **关系型** — 依赖人际信任、沟通协调、共识建立
6. **设计型** — 需要从零或从约束条件出发设计方案/机制/架构
7. **混合型** — 上述多类叠加，不可简单归为单一类型

---

## S2 工作颗粒度拆解

把工作对象拆成可判断、可推演、可比较的最小分析单位。

每个颗粒度必须包含以下 8 个字段：
1. **名称** — 短标签（如"异常数据识别与通知"）
2. **定义** — 一句话说清这个颗粒度做什么
3. **典型场景** — 在什么情况下发生
4. **当前主要角色** — 现在主要由谁/什么角色承担
5. **是否高频** — 是/否
6. **是否高价值** — 是/否（对核心目标有直接撬动）
7. **是否强依赖经验** — 是/否（新手能否快速上手）
8. **是否跨角色/跨部门** — 是/否

---

## S3 问题处理模式

判断这个颗粒度的问题，本质上应该怎样被处理：

1. **规则判定型** — 求明确结论（有标准答案或可计算）
2. **风险识别型** — 求风险等级、概率、趋势判断
3. **归因诊断型** — 求原因解释、因果拆解
4. **方案设计型** — 求可行动方案、机制设计、路径选择
5. **共识收敛型** — 求多角色判断对齐、利益协调

---

## S4 执行主体重构

判断未来这项工作主要由谁来做：

1. **自动接管** — AI/系统直接完成，人仅做异常审核
2. **AI主辅协同** — AI 先做一层（初筛/初稿/初判），人再把关和升级
3. **人工主导，AI辅助** — 人做核心判断，AI 提供信息/建议/选项
4. **人工主导，AI仅弱辅助** — AI 基本不介入核心环节

---

## S5 价值增益方式

判断 AI 介入后的主要价值提升方向：

1. **提效** — 更快、更省人力
2. **提质** — 更准、更稳、更少出错。在高争议场景中，也包括"组织接受度提升""争议减少"
3. **扩容** — 覆盖更多对象/场景/频次
4. **洞察** — 看见过去看不见的模式、趋势、关联
5. **设计** — 辅助生成和评估方案/机制/架构

---

## S6 AI 介入路径与现实约束

### S6-A 基础能力层（AI 靠什么能力介入）

| 代码 | 能力 | 说明与示例 |
|------|------|-----------|
| A | 读取理解 | 文本/图片/音频/视频的内容理解与信息提取 |
| B | 规则流程 | 按预设规则/流程执行判断、推进、触发 |
| C | 检索召回 | 从大规模信息中检索相关内容并排序 |
| D | 分类分级 | 将输入按标准分类或评等 |
| E | 生成表达 | 生成文本/图片/代码/方案等内容 |
| F | 推理比较 | 多选项间的逻辑推理、比较、推断 |
| G | 交互协同 | 多步骤人机对话、追问、澄清、确认 |
| H | 外部行动 | 调用外部 API/工具/系统执行操作 |

### S6-B 现实约束层

对每个颗粒度评估 4 个约束维度：

1. **技术成熟度** — 高/中/低：当前 AI 能力是否足以胜任
2. **市场供给状态** — 充足/有限/空白：是否有现成工具/方案
3. **成本等级** — 低/中/高：实施和运行成本
4. **启动主体** — 个人可启动 / 团队可启动 / 需要组织级投入

### S6-C 扩展动作层：GitHub 开源参考实现检索

> 这是通用版的核心差异化能力。不是被动回答"能不能做"，而是主动帮用户找到"已经有谁做了类似的东西"。

#### 触发条件（Gate Check）

对每个工作颗粒度，同时满足以下条件才触发搜索：
- S4 标签 ∈ {自动接管, AI主辅协同}
- S6-B 至少 2 项评估为有利（技术成熟度≥中 + 市场供给≠空白 + 成本≤中）

#### 执行流程

1. **特征提取**：从颗粒度记录中提取名称、定义、痛点、S6-A 能力关键词
2. **查询构建**：并行构建 3 类搜索词（中文语义 / 英文技术 / 能力关键词）
3. **搜索执行**：通过 WebSearch 工具执行搜索，跨查询结果去重
4. **成熟度评分**：见 `references/core/github-search-protocol.md` 评分矩阵
5. **推荐呈现**：Top 3-5 作为推荐卡片输出（项目名+映射+成熟度+采纳难度+路径）

#### 评分维度（详见 github-search-protocol.md）

- Stars/Fork 数量
- 最近提交时间
- 文档完整度
- 社区活跃度

---

## 框架使用顺序

S1 → S2 → S3 → S4 → S5 → S6-A → S6-B → (Gate) → S6-C

**不可跳步、不可乱序。** 每一层判断依赖上一层的输出。

---

## 核心原则

1. 先拆工作颗粒度，再谈 AI 影响
2. 先判断问题处理模式（S3），再讨论谁来做（S4）
3. 不要对本质上不确定的问题强行给出确定答案
4. 低阶执行工作最容易被接管，高阶设计工作最容易发生价值重构
5. 不是所有变化都等于替代，很多变化发生在工作形态本身
6. 一旦某颗粒度需要多角色/多部门共同判断，就把它视为跨模块关键议题
7. 工作未来命题必须收敛成"从什么转向什么"的句式
8. 对高 AI 可行性的颗粒度，主动检索 GitHub 开源实现，打通"应该做什么"到"用什么做"
